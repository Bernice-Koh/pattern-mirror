# Conventions

Engineering conventions for pattern-mirror. Language-level code style lives in [CODE_STYLE.md](CODE_STYLE.md); this file is about *how we work*, not how we write code.

The project shape these assume: a Python/FastAPI backend with a LangGraph-orchestrated analysis engine calling the Anthropic API directly, a React + TypeScript frontend, PostgreSQL, and Docker Compose locally with Azure Container Apps as the eventual production target. Solo developer, four-week MVP window, weekly sprints.

## Folder structure

Four concerns at the root: `backend/`, `frontend/`, `docs/`, `deploy/`. Never mix.

Backend code is one installable package, `pattern_mirror`, under `backend/src/` (a standard src-layout). Inside `backend/src/pattern_mirror/`, separated by what code is *for*, not by file type:

- `api/` — FastAPI routers and request/response models. Thin.
- `engine/` — the five-stage analysis pipeline and LangGraph orchestrator. Stage naming follows the [design spec](DESIGN_SPEC.md): intelligent components are *Agents*, deterministic components are *Modules*.
- `services/` — business logic callable from API handlers, jobs, and tests alike.
- `models/` — SQLAlchemy ORM models.
- `db/` — engine/session setup, Alembic migrations, raw queries if any.
- `jobs/` — entrypoints for scheduled work (e.g. nightly dictionary-candidate summarisation).
- `core/` — config, typed exceptions, logging setup.

Tests live in `backend/tests/`, a parallel tree mirroring the package's purpose folders (e.g. `pattern_mirror/api/health.py` → `tests/api/test_health.py`). Create folders when there's code to put in them, not before.

## Dependencies and tooling

- **Python:** 3.12, managed with `uv`. Deps in `backend/pyproject.toml`; `backend/uv.lock` is committed. `uv add <pkg>` to add, `uv run <cmd>` to run.
- **Frontend:** `npm`; `frontend/package-lock.json` is committed.
- **Python lint/format:** `ruff` for both.
- **Python types:** `mypy --strict` against `backend/src/`.
- **Frontend lint/format:** `eslint` + `prettier`.
- **Pre-commit:** the `pre-commit` framework runs lint/format/type checks before every commit.
- **CI:** GitHub Actions runs lint, type-check, and tests on every PR. CI must pass before merge.

## Database, ORM, migrations

- **Engine:** PostgreSQL 16. Local dev and tests use the container in `deploy/docker-compose.yml`; tests must work offline.
- **ORM:** SQLAlchemy 2.x typed declarative API. Pydantic models are separate from ORM models; never use one for both roles.
- **Migrations:** Alembic, one revision per schema change. Autogenerated revisions are read and edited before committing. The migration history — not the model classes — is the schema source of truth.
- **Blobs:** document content goes to blob storage (Azure Blob in production; a local-disk or Azurite stand-in for dev), never into Postgres as large text columns. Postgres holds metadata and blob references.

## Testing

- `pytest` for Python, `vitest` for the frontend.
- Tests mirror the `src/` layout. One behaviour per test.
- Tests run migrations against the local docker-compose Postgres — never SQLite as a stand-in. Spin up clean, migrate, run, tear down.
- **Mock all Anthropic API calls in tests.** Deterministic fixtures for Contextual Pass, Judge, and Recommendations outputs. Tests never hit the live API.
- **Gold set.** Maintain a small labelled set of documents (JDs, feedback, writeups) with known-correct flags from early on. Engine calibration — precision/recall per stage, agreement rate — is measured against it, not assumed. The gold set is also the demo's seed data and the calibration metric in the [design spec](DESIGN_SPEC.md).
- The deterministic stages (Dictionary Service, Adjudicator, fingerprinting/suppression logic) get thorough unit tests — they're cheap to test and carry the system's correctness guarantees.

## Logging

- Structured JSON via `structlog`. Never `print`.
- Every log line carries a correlation ID propagated through the request or engine run. Engine stages log entry/exit with the flag or document ID — the audit trail is a product feature here, not just ops hygiene.
- `DEBUG` tracing, `INFO` normal flow, `WARNING` unexpected-but-recoverable, `ERROR` failures.

## Error handling

- Typed exceptions in `core/errors.py`, all subclassing one project base error.
- API routes catch at the handler boundary only; engine stages raise, the orchestrator decides whether a stage failure kills the run or degrades it (e.g. LLM pass fails → dictionary flags still return).
- Never swallow exceptions silently.

## Git

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:`.
- Subject ≤ 72 chars, imperative mood. Body explains *why* when not obvious.
- One concern per commit. Stage explicit paths; never `git add .`.

## Pull requests

- Branch names: `feat/<short-description>`, `fix/<...>`, `chore/<...>`, `spike/<...>`.
- Every PR references its issue (`Closes #12`) and says what changed, why, and how to test. A screenshot for any UI change.
- Keep PRs small enough to review in one sitting — if a PR mixes two stories, it should have been two PRs.
- CI must pass before merge. Squash on merge; `main` stays linear and protected.

## Project management

- GitHub Projects with a 1-week iteration field is the sprint mechanism. Milestones track external programme deadlines (Huddle 3, Closing Ceremony); iterations track internal cadence.
- Epics are labels (`epic:*`); stories and tasks are issues with `type:*` labels; tasks for the active sprint live as checklists inside their story.
- Only the next sprint is fully refined. The backlog beyond it stays title-only by design.
- Board hygiene is part of finishing work: a story isn't done until its card is in Done and the PR is merged.

## Secrets

- Never commit secrets, even temporarily. `.env` is gitignored; `.env.example` (template only) is committed.
- Each service owns its own config — `backend/.env` alongside `backend/.env.example`, and the frontend likewise — loaded relative to that service's directory. There is no shared root `.env`.
- The Anthropic API key exists only in `.env` locally and in CI/deploy secret stores — never in code, issues, or logs.
- Synthetic data only: no real UBS documents, employee names, or feedback ever enter this repo. MVP peer-feedback data is mocked by design.
