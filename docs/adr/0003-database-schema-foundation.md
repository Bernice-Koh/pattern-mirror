# 3. Database schema foundation

- Status: Accepted
- Date: 2026-06-18

## Context

Issue #17 builds the persistence foundation: local Postgres via docker-compose,
SQLAlchemy 2.x typed models, Alembic, and the first migration covering the core
entities. The logical schema and its modelling rationale (decisions D1–D11) were
settled in the issue #17 schema design brief; this ADR records the
implementation choices made while turning that brief into code, not the schema
rationale itself (the brief and CLAUDE.md's "Decisions made and locked" hold
that).

Scope note: the AC clause "the engine selects region rows at runtime" is the
orchestrator's responsibility in a later issue. This PR delivers the *enabling*
schema — `dictionaries.region_code`, the `(region_code, active)` lookup index
shaped for that predicate, and the seeded regions — not the selection code
itself.

## Decision

- **Migration 0001 creates all 11 foundation tables**, per the brief's roadmap:
  `regions`, `citations`, `dictionaries`, `users`, `user_roles`, `subjects`,
  `documents`, `analysis_runs`, `flags`, `flag_dismissals`, `agent_runs`. This is
  the engine core plus JD Studio and satisfies the acceptance criteria. Tables
  for later features (`recommendations`, `drift_findings`, `peer_feedback`,
  `dictionary_candidates`) land in migrations 0002–0005 with the features that
  use them.

- **Models mirror migration 0001 exactly.** Two columns the brief assigns to
  migration 0005 — `dictionaries.origin_candidate_id` and
  `agent_runs.candidate_id` — are omitted here because they reference the
  `dictionary_candidates` table that 0005 introduces. Keeping the models and the
  migration in lockstep is what lets `alembic check` act as a parity guard.

- **`alembic check` is the verification gate, not blind autogenerate.** The
  migration was drafted with `--autogenerate`, then hand-finalised; `alembic
  check` reports no pending operations, proving the migration and the models
  agree. A metadata-wide naming convention (`db/base.py`) gives every constraint
  and index a deterministic name so that comparison is stable across machines.

- **Native enum types are created once, explicitly, with `create_type=False`.**
  The shared `bias_category` type is referenced by both `flags` and
  `dictionaries`; left to implicit DDL, the second table's `CREATE TYPE` would
  fail. The migration creates all 13 enum types up front (and drops them in
  `downgrade`), and the model-side enum instances are `create_type=False` so only
  the migration owns enum DDL.

- **`subjects.gender` is `VARCHAR`, not a native enum.** Alembic autogenerate
  cannot detect enum *value* changes, and gender is a set that may need to grow;
  a string avoids a hand-written `ALTER TYPE` migration later. The fixed sets
  (`doc_type`, `bias_category`, verdict, etc.) stay native enums.

- **Sequential revision ids (`0001_foundation`)** map directly to the brief's
  migration roadmap; generate later ones with `alembic revision --rev-id 000N`.

- **Seed scope for 0001 is `regions` only** (the SEA jurisdiction set; only `SG`
  carries a lexicon in MVP). Citation, dictionary, user, and subject seeds belong
  with the engine and feature work that consumes them, and keep this PR
  reviewable in one sitting.

- **Settings gain `database_url` (required) and `test_database_url` (optional).**
  URLs are plain strings, not `PostgresDsn`, so the `+psycopg` driver suffix
  passes through to `create_engine` untouched.

- **Test isolation.** docker-compose provisions `pattern_mirror_test` alongside
  the dev database via an init script on first boot. A session-scoped fixture
  runs `alembic upgrade head` against the test database; each `db_session` runs
  in a transaction rolled back afterwards, so tests neither persist data nor
  touch the dev database. When no Postgres is reachable, the database tests skip
  rather than fail, keeping the non-database suite runnable offline.

## Consequences

- **+** A clean machine reaches the full schema with `docker compose up` then
  `alembic upgrade head`; the test database is migrated automatically by the
  suite.
- **+** `alembic check` in CI will catch any future drift between models and
  migrations.
- **+** Adding a SEA market is an `INSERT` into `regions`, not a migration.
- **−** Enum DDL is hand-managed in migrations rather than fully autogenerated;
  this is the deliberate cost of native enums shared across tables.
- **−** Database tests require a reachable Postgres; they skip (rather than run
  against a SQLite stand-in) when one is absent, so a fully offline run does not
  exercise the schema.

## Alternatives considered

- **`managers` + `hr_users` split / `documents` per type / storing patterns** —
  rejected in the schema brief (D1, D2, D7); not reopened here.
- **Native enum for `gender`** — rejected for the enum-value-migration cost above.
- **Autogenerate the migration verbatim** — its implicit enum creation fails for
  the shared `bias_category` type; the migration is hand-finalised and verified
  with `alembic check` instead.
- **conftest creates the test database on demand** — rejected in favour of the
  docker init script, so provisioning lives in one place and the test harness
  only migrates.
- **Random Alembic revision hashes** — replaced with sequential ids that match
  the roadmap, since the history is linear and small.
